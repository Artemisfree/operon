# MVP Readiness Plan

## Цель
Подтвердить, что проект готов к MVP-показу и пилотному использованию по основному сценарию:
`Website Chat -> Order -> Handoff -> Delivery -> Review`

## Scope
В план входят только обязательные работы для MVP:
- сквозная проверка основного сценария
- проверка критериев приёмки MVP
- устранение критичных багов
- ручная проверка frontend-приложений
- финализация документации и runbook
- фиксация итогового статуса готовности

---

## 1. Сквозной E2E Smoke

### Цель
Подтвердить, что весь основной путь работает как единый продукт.

### Сценарий
1. Клиент открывает `website-widget`
2. Клиент оформляет заказ через AI-чат
3. AI создаёт заказ
4. Оператор открывает диалог в `admin-web`
5. Оператор включает handoff
6. Оператор отправляет сообщение вручную
7. Оператор возвращает управление AI
8. Заказ переводится в статус доставки
9. Диспетчер назначает курьера
10. Курьер открывает `courier-web`
11. Курьер переводит доставку в `on_the_way`
12. Курьер загружает proof photo
13. Курьер переводит доставку в `delivered`
14. Система создаёт и/или отправляет review request
15. Метрики и итоговые статусы доступны в `admin-web`

### Артефакты
- e2e smoke checklist
- smoke script или ручной сценарий
- список ожидаемых результатов на каждом шаге

### Definition of Done
- весь сценарий проходит без ручных SQL-правок
- нет блокирующих сбоев
- все ключевые статусы отражаются в UI и в API

---

## 2. Проверка Критериев Приёмки MVP

### Цель
Сверить текущую реализацию с исходными критериями MVP.

### Критерии
1. Клиент оформляет заказ через website chat
2. AI использует tools и не выдумывает каталог
3. Оператор перехватывает чат одной кнопкой
4. Диспетчер назначает курьера
5. Курьер закрывает доставку
6. Через 5–10 минут после `delivered` создаётся или отправляется review request
7. Сквозной сценарий проходит стабильно

### Задачи
- составить таблицу `критерий -> способ проверки -> статус`
- отметить `done / partial / blocked`
- собрать ссылки на API/UI, которыми это подтверждается

### Definition of Done
- все критерии имеют явный статус
- по каждому `partial` или `blocked` есть причина и план добивки

---

## 3. Финальная Стабилизация Backend

### Цель
Убрать критичные сбои в основном MVP-потоке.

### Задачи
- проверить запуск с нуля:
  - `docker compose up`
  - миграции
  - сиды
- проверить real OpenAI mode
- проверить guardrails:
  - без обязательных полей заказ не создаётся
  - при handoff AI не отвечает
- проверить review flow:
  - не создаёт дубликаты
  - корректно отрабатывает задержку
- проверить delivery workflow:
  - `assign -> on_the_way -> delivered`
  - proof photo сохраняется

### Приоритет фиксов
- `P0`: ломает основной путь MVP
- `P1`: даёт неверное состояние заказа/доставки/review
- `P2`: не блокирует MVP-показ, можно оставить в backlog

### Definition of Done
- все `P0` и `P1` баги исправлены
- backend стабильно проходит smoke

---

## 4. Ручная Проверка Frontend

### Цель
Убедиться, что UI пригоден для MVP-демо и базового пилота.

### Website Widget
- открывается без ошибок
- отправляет сообщение
- показывает AI-ответ
- показывает handoff state
- обновляет историю через polling

### Admin Web
- login работает
- список диалогов отображается
- карточка диалога открывается
- handoff start/stop работает
- operator reply работает
- список заказов и order workflow доступны
- назначение курьера доступно
- метрики отображаются

### Courier Web
- вход по токену работает
- список назначенных доставок отображается
- смена статуса работает
- загрузка proof photo работает
- закрытие доставки работает

### Definition of Done
- нет блокирующих UI-багов
- все три приложения проходят ручной smoke

## 5. Runbook И Документация

### Цель
Сделать MVP воспроизводимым без устного контекста.

### Нужно задокументировать
- как поднять проект
- как применить миграции
- как загрузить сиды
- какие тесты запускать
- какие dev URL используются
- какие demo credentials использовать
- как прогнать MVP smoke
- какие известные ограничения остаются

### Файлы
- `README.md`
- `IMPLEMENTATION_PLAN.md`

### Definition of Done
- новый участник может поднять MVP по документации
- smoke можно прогнать по инструкции шаг за шагом

---

## 6. Финальный MVP Gate

### Цель
Принять решение о готовности MVP.

### Возможные статусы
- `MVP Ready`
- `MVP Ready with Known Limitations`
- `Not Ready`

### Формат решения
- итоговый статус
- список блокеров
- список неблокирующих ограничений
- список пост-MVP задач

### Definition of Done
- статус MVP зафиксирован в документации
- есть явный список того, что ещё можно делать после релиза MVP
