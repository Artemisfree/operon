# План реализации конструктора на базе AI-агента (Website, MVP)

## Принятые решения (по вашим ответам)
- Делаем **полный цикл**: intake → processing → dispatch/tracking → review request.
- Канал на MVP: **только Website** (без Telegram/WhatsApp на старте).
- LLM-провайдер: **OpenAI**.
- Для MVP делаем **упрощённую архитектуру** (без тяжёлых enterprise-фич).
- Каталог и настройки: через **админку**.
- Онлайн-платежи: **не включаем в MVP**.
- Доставка: **свои курьеры**.
- Compliance/сложные требования безопасности: **минимально необходимый базовый уровень**.
- Язык: **только русский**.

## Рекомендация по handoff (п.4)
Для MVP оптимально сделать **полный ручной перехват диалога оператором** (simple handoff):
- одна кнопка «Перехватить чат»;
- после перехвата AI не отправляет сообщения, пока оператор не вернёт чат AI;
- сохраняется история действий AI/оператора.

Почему так лучше для MVP:
- проще и быстрее реализовать;
- меньше рисков конфликтов «AI + оператор одновременно»;
- проще объяснить и обучить команду.

## Целевая функциональность MVP (website-first, AI-agent core)

### 1) Intake (Website chat)
- Виджет чата на сайте.
- AI-агент приветствует клиента, отвечает на вопросы, предлагает товары из каталога.
- Сбор данных для заказа: товар, количество, адрес, контакт.
- Эскалация к оператору через handoff.

### 2) Business data (внутри админки)
- Сущности:
  - `products` (каталог),
  - `customers`,
  - `conversations`,
  - `orders`,
  - `order_status_history`,
  - `delivery_jobs`,
  - `review_requests`.
- Один простой tenant (single-tenant MVP) с возможностью расширения позже.

### 3) Order processing (операторская панель)
- Таблица заказов со статусами: `new`, `confirmed`, `preparing`, `ready_for_delivery`, `out_for_delivery`, `delivered`, `cancelled`.
- Карточка заказа: состав, адрес, комментарии, ответственный.
- Ручные действия оператора: подтвердить/отменить/изменить статус.
- Добавление proof (фото) при завершении доставки.

### 4) Dispatch & tracking (для своих курьеров)
- Экран диспетчера: назначение заказа курьеру.
- Экран курьера (mobile-first web): список активных доставок.
- Курьер обновляет статусы: `accepted` → `on_the_way` → `delivered`.

### 5) Review request
- Автоматическое создание задачи запроса отзыва через 5–10 минут после `delivered`.
- Для MVP: отправка review request в website-канал (или сохранение задачи на отправку оператором, если клиент офлайн).

## Технический план реализации

## Этап 1. Базовый каркас (Week 1)
- Backend API + БД + миграции.
- Модели данных для чатов, заказов, доставки.
- Простая админка (логин, каталог, заказы).

## Этап 2. AI-агент на OpenAI (Week 1–2)
- Интеграция OpenAI API.
- Системный промпт для RU-сценария заказа.
- Tool-calling к функциям:
  - поиск товара,
  - создание заказа,
  - получение статуса заказа.
- Логи запросов/ответов AI (для отладки).

## Этап 3. Website chat + handoff (Week 2)
- Виджет чата на сайт.
- Передача сообщений в backend и агенту.
- Кнопка «Перехватить чат» в панели оператора.
- Режим возврата чата обратно AI.

## Этап 4. Процессинг заказов + доставка (Week 3)
- Workflow статусов в панели.
- Назначение курьера.
- Курьерский mini-web интерфейс.
- Загрузка фото-подтверждения доставки.

## Этап 5. Review automation + стабилизация (Week 4)
- Отложенный триггер review request (5–10 минут).
- Минимальная аналитика: количество заказов, % доставленных, % handoff.
- Полировка UX, исправление багов, подготовка к пилоту.

## Что сознательно НЕ делаем в MVP
- Telegram/WhatsApp интеграции.
- Онлайн-оплату.
- Сложную мультитенантность.
- Продвинутый RBAC и enterprise compliance.
- Сложные AI-многоагентные сценарии.

## Критерии готовности MVP
- Клиент может оформить заказ через website-чат с участием AI.
- Оператор может перехватить любой чат и завершить заказ вручную.
- Диспетчер назначает курьера, курьер обновляет статусы доставки.
- После доставки запускается review request через 5–10 минут.
- Вся цепочка работает на русском языке.
